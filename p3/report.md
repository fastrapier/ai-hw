## Наблюдаемая система

В качестве наблюдаемой системы выбран ИИ-агент из `p2`.

Он реализован как endpoint `POST /agent/answer` и работает по следующей схеме:

1. принимает вопрос пользователя;
2. выбирает локальный контекст по проекту;
3. формирует системный промпт и пользовательский prompt;
4. отправляет запрос в локальную LLM через Ollama;
5. возвращает ответ и метаданные контекста.

Такой вариант хорошо подходит для LLMOps, потому что в нём уже есть типичный агентный пайплайн: вход, подбор контекста, генерация, обработка ошибок, зависимость от локальной модели.

## Сравнение observability-стеков

Для выбора стека были рассмотрены три варианта.

### 1. Langfuse

Плюсы:

- open-source решение, ориентированное именно на LLM-приложения;
- есть traces, generations, metadata по промптам и ответам;
- удобный UI для короткой демонстрации;
- подходит для кастомных Python-приложений, не только для готовых фреймворков.

Минусы:

- требует дополнительной инфраструктуры;
- для self-hosted варианта нужен более тяжёлый стек, чем у простой метрики или обычного логирования.

### 2. OpenLLMetry / OpenTelemetry

Плюсы:

- стандартизованный подход к телеметрии;
- гибкость и хорошая интеграция с разными backends;
- полезен как более общий observability-подход.

Минусы:

- для учебного LLM-демо нужен больший объём ручной настройки;
- без дополнительного UI и интеграций сложнее показать именно prompt/output-centric наблюдаемость.

### 3. Классический стек Prometheus + Loki + Jaeger

Плюсы:

- хорошо закрывает общие логи, метрики и трейсы;
- широко используется в backend-инфраструктуре;
- даёт сильную базу по production observability.

Минусы:

- стек ориентирован на общую сервисную наблюдаемость, а не на LLM-специфику;
- для анализа промптов, ответов и агентных шагов он менее удобен, чем Langfuse.

## Почему выбран Langfuse

`Langfuse` был выбран как основной стек, потому что:

- он лучше всего подходит для наблюдаемости именно LLM и агентных пайплайнов;
- позволяет быстро показать traces и generations в локальном UI;
- даёт более понятную демонстрацию prompt/response/debugging, чем более общий observability-стек;
- хорошо сочетается с уже существующим агентом из `p2`.

## Что именно было проинструментировано

Интеграция добавлена вокруг `POST /agent/answer`.

В trace сохраняются:

- вопрос пользователя;
- выбранные агентом источники контекста (`selected_tools`, `context_sources`);
- запрошенная и фактическая модель;
- итоговый prompt и system prompt;
- текст ответа модели;
- успех или ошибка выполнения.

Внутри одного запроса создаются наблюдения:

- `agent-answer` - корневая операция;
- `context-selection` - выбор локального контекста;
- `llm-generation` - генерация ответа через Ollama.

## Логи, метрики и трейсы

### Логи

Логи приложения и контейнеров выводятся в stdout/stderr и доступны через Docker logs. Это даёт базовый operational layer для диагностики падений и ошибок инфраструктуры.

### Метрики

Для LLMOps-сценария наиболее полезны следующие метрики:

- доля успешных agent-запросов;
- latency ответа;
- количество ошибок генерации;
- стабильность выбранной модели;
- LLM-специфичные показатели, например usage/response-level metadata, если они доступны в backend.

### Трейсы

Langfuse хранит трассы запросов и позволяет увидеть:

- структуру agent-запроса;
- последовательность наблюдений;
- входы и выходы каждого шага;
- связь между выбором контекста и итоговым ответом.

## Локальный runtime

Для воспроизводимого локального запуска был собран Docker Compose стек в `p3/docker-compose.yml`.

Он включает:

- API из `p2`;
- локальный Ollama;
- Langfuse Web;
- Langfuse Worker;
- PostgreSQL;
- ClickHouse;
- Redis;
- MinIO.

Это позволяет локально поднять наблюдаемость без облачных сервисов и показать работу на собственной машине.

## Демонстрационный сценарий

Базовый сценарий показа:

1. поднять стек;
2. отправить осмысленный запрос в `POST /agent/answer`;
3. открыть Langfuse UI;
4. показать trace, prompt metadata, selected context и ответ модели.

Примеры вопросов:

- `Какая модель выбрана для агента и почему?`
- `Какие уязвимости есть у агентного решения?`
- `Какие endpoint'ы есть в FastAPI API проекта?`

## Какие отказы и сбои помогает обнаруживать observability

- недоступность Ollama;
- деградация latency генерации;
- неудачные или слишком общие ответы модели;
- ошибки в подборе контекста;
- различия между ожидаемым и фактическим поведением агента на реальных запросах.

## Ограничения решения

- self-hosted Langfuse требует заметно больше инфраструктуры, чем простой backend без observability;
- часть полезных метрик зависит от того, какие данные возвращает конкретный LLM backend;
- наблюдаемость не устраняет галлюцинации автоматически, а только помогает быстрее увидеть и анализировать проблемные случаи.
