## 1. Текст — тональность (Hugging Face)

**Библиотека:** `transformers`, backend **PyTorch** (CPU).  
**Плюсы для задачи:** готовый высокоуровневый `pipeline("sentiment-analysis")`, быстрый старт, множество предобученных чекпоинтов на Hugging Face Hub, единый API для разных моделей.

**Модель:** `cardiffnlp/twitter-xlm-roberta-base-sentiment` — дообученный **XLM-RoBERTa** на данных Twitter; поддерживает **несколько языков**, что удобно для русско-английских текстов. Выбор обоснован балансом качества и простоты вызова без собственного обучения.

**Принцип работы:** токенизация входа → энкодер Transformer → линейный классификационный слой по меткам тональности; выход — метка и вероятность (softmax по классам головы).

**Данные / датасет:** в демо используются **пользовательские** тексты в `text/input/*.txt` (пример `sample.txt`). Это не фиксированный бенчмарк; структура входа — **plain text**, выход — JSON с полями `label`, `score`.

**Метрики качества:**

- **ML:** на полноценной оценке использовали бы accuracy / F1 на размеченном корпусе; в рамках демо — **санити-чек**: осмысленные полярности на явно позитивных/негативных фразах, согласованность `score` как уверенности.
- **Продуктовые:** латентность ответа на один короткий текст (время end-to-end в контейнере).
- **Инфраструктурные:** размер скачанных весов на Hub, использование CPU/RAM контейнера, повторный запуск с тем же volume кэша (без повторной загрузки).

**Особенности реализации:** усечение до 512 токенов; переменные `INPUT_DIR`/`OUTPUT_DIR`; результат в `output/result.json`. Задача выбрана как **кандидат под будущий HTTP API** (простой JSON ввод-вывод).

## 2. Аудио — YAMNet (TensorFlow + TensorFlow Hub)

**Библиотека:** **TensorFlow**, **TensorFlow Hub**.  
**Плюсы:** стабильная загрузка SavedModel с Hub, оптимизированный граф инференса, хорошая документация YAMNet; отдельный стек от HF/PyTorch в других задачах (выполнение требования «≥2 фреймворков»).

**Модель:** **YAMNet** (`yamnet/1` на TF Hub) — классификация на **521** класс AudioSet (аудио-события).

**Принцип работы:** вход — волна **16 kHz** mono; внутри — лог-мел спектрограмма и свёрточная сеть; на выходе — распределение по классам во времени; в демо усредняются скоры по времени и берётся **top-5**.

**Данные:** WAV в `audio/input/` (или **синтетический** синус, если файла нет). Имена классов подгружаются из CSV **yamnet_class_map** (нужен сетевой доступ при первом запуске для загрузки таблицы классов).

**Метрики:**

- **ML:** на бенчмарке — mAP / top-k accuracy по AudioSet; в демо — правдоподобность топ-классов на реальном звуке, на синтетике — только проверка **работоспособности** пайплайна.
- **Инфраструктурные:** время загрузки Hub-модуля, объём `TFHUB_CACHE_DIR`.

**Особенности:** ресемплинг к 16 kHz в `run.py`; переменная `TFHUB_CACHE_DIR` согласована с volume в compose.

## 3. Изображения — ViT-B/16 ImageNet (PyTorch / torchvision)

**Библиотека:** **PyTorch**, **torchvision**.  
**Плюсы:** веса ImageNet в одном API (`ViT_B_16_Weights`), метаданные классов в `weights.meta["categories"]`; ViT и **усреднение по TenCrop** дают более устойчивые предсказания на крупных сценах, чем один center crop у ResNet-50.

**Модель:** **ViT-B/16** с весами **IMAGENET1K_V1**; инференс: 10 кропов, среднее логитов, затем softmax.

**Принцип работы:** изображение режется на патчи, Transformer над последовательностью патчей, классификационная голова на 1000 классов ImageNet. Ограничение: ImageNet в основном **натуралистичные фото**; **живопись** и репродукции часто дают низкие вероятности и «странные» топ-метки.

**Данные:** изображения в `image/input/` (поддерживаемые расширения); при отсутствии — **синтетический** градиент 224×224 (проверка кода, топ-классы условны).

**Метрики:**

- **ML:** на ImageNet — top-1 / top-5 error; в демо на **фото** — визуальная проверка топ-1; на **картинах** и плейсхолдере — ожидаемо слабая калибровка.
- **Инфраструктурные:** время первой загрузки весов в `TORCH_HOME` / volume `torch_cache` (ViT тяжелее ResNet-50).

## 4. Видео — выборка кадров + DETR (Hugging Face)

**Библиотека:** **transformers**, **PyTorch** (CPU), **OpenCV** (чтение/запись видео).  
**Плюсы:** готовая постобработка боксов в `DetrImageProcessor.post_process_object_detection`, модель DETR на COCO из коробки.

**Модель:** **DETR ResNet-50** (`facebook/detr-resnet-50`) — детекция объектов (**COCO**), Transformer over CNN backbone.

**Принцип работы:** из потока кадров выбираются отдельные кадры; каждый кадр обрабатывается как изображение; модель предсказывает классы и ограничивающие прямоугольники; фильтрация по порогу `SCORE_THRESHOLD`.

**Данные:** видео в `video/input/`; иначе генерируется короткий **AVI** (MJPEG) средствами OpenCV в контейнере.

**Метрики:**

- **ML:** на COCO — mAP; в демо — **качество на глаз** (есть ли разумные боксы на реальном ролике), на синтетике — в основом проверка отсутствия падений и формата JSON.
- **Продуктовые:** время обработки **N** кадров.
- **Инфраструктурные:** пик RAM при загрузке DETR, использование общего `hf_cache` с текстовой задачей.

## 5. Локальная LLM (Ollama)

**Инструмент:** официальный образ **Ollama** в Docker, volume для данных.  
**Плюсы:** простая загрузка моделей (`ollama pull`), локальный REST API, независимость от облачных провайдеров (соответствие формулировке задания).

**Модель (пример):** `llama3.2:3b` или другая **3B–8B** из каталога Ollama — в зависимости от терпимости к времени ответа на CPU.

**Принцип:** сервер загрузки и инференса GGUF/квантованных весов (реализация на стороне Ollama); клиент — CLI или HTTP API.

**Метрики:**

- **Продуктовые:** время до первого токена / полного ответа, субъективное качество ответа на короткий промпт.
- **Инфраструктурные:** размер модели на диске (volume), загрузка CPU, стабильность сервиса при `docker compose up`.

Подробные команды: [llm/README.md](./llm/README.md).
